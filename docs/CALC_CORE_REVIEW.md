# Ревью калькуляционного ядра (Задача 37)

**Обновление:** рекомендации из ревью реализованы: ограничение principal по balance, единая методология процентов (ACTUAL_365 через график), расчёт effectiveInterestRate, округление до копеек, юнит-тесты (см. `src/utils/*.test.ts`).

## Обзор архитектуры

Цепочка расчётов:

1. **Форма** (`LoanDetailsForm` → `useLoanForm`) → `unformatFormValues` → числа и даты
2. **Провайдер** (`MortgageProvider`) получает `loanDetails` и при изменениях вызывает:
   - `mortgageService.calculateBase(loanDetails)` → базовые результаты (без досрочных)
   - `mortgageService.generateAmortizationSchedule({ ...loanDetails, earlyPayments, regularPayments })` → график и сводка
3. **Ядро расчётов**:
   - `mortgageCalculator.ts` — базовая формула: платёж, сумма процентов, дата окончания
   - `financialMath.ts` — формулы аннуитета, дифференцированного платежа, процентов за период, даты
   - `amortizationSchedule.ts` — помесячный график с учётом досрочных и регулярных доплат

## 1. Корректность формул

### Аннуитет

- **Формула**: `M = P * (r * (1+r)^n) / ((1+r)^n - 1)` в `financialMath.calculateMonthlyPayment`.
- Реализация соответствует стандартной формуле аннуитета. **Верно.**

### Дифференцированный платёж

- Фиксированная часть основного долга: `principal / numberOfPayments`.
- Проценты за месяц: `remainingPrincipal * monthlyRate`.
- Реализация в `calculateDifferentiatedMonthlyPayment` корректна. **Верно.**

### Итоговая стоимость по типу платежа

- Аннуитет: `monthlyPayment * loanTerm * 12` — верно.
- Дифференцированный: сумма помесячных платежей в цикле — верно (см. ниже про оптимизацию).

### Проценты за период (график)

- `calculateInterestForPeriod` с методом ACTUAL_365:  
  `balance * (annualRate/100) * (days / daysInYear)` — корректная схема «факт/365».

### Дата окончания

- `calculatePayoffDate`: к дате начала добавляются годы срока — логично для «базового» сценария без досрочных.

## 2. Несогласованность: база vs график

- **Базовый расчёт** (`mortgageCalculator` + `financialMath`) считает проценты по **месячной ставке** `rate/12` (равные месяцы).
- **График** (`amortizationSchedule`) считает проценты по **ACTUAL_365** (реальное число дней в периоде).

Из-за этого:

- `totalInterest` из `calculateBase()` и сумма процентов по графику (без досрочных) **не совпадут**.
- В интерфейсе «без досрочных» показывается одно число, а в графике по месяцам — другое.

**Рекомендация:** либо везде перейти на одну модель (например, ACTUAL_365 и в базе считать через «виртуальный» график без досрочных), либо в базе тоже считать проценты по дням и документировать, что база и график используют одну и ту же методологию.

## 3. Ошибки и граничные случаи в графике

### 3.1. Последний платёж / малый остаток (principal > balance)

- Для **дифференцированного** платежа:  
  `principal = fixedPrincipalPortion = loanAmount / numberOfPayments` не ограничивается остатком.
- Для **аннуитета**: `principal = currentMonthlyPayment - interest` тоже может стать больше `balance` на последнем шаге.
- Сейчас: `newBalance = balance - principal` может уйти в минус, в выводе используется `Math.max(0, newBalance)`, но в самой строке графика `principal` и `payment` остаются завышенными, а следующий цикл выходит по `balance > 0.01`.

**Рекомендация:** перед записью строки графика ограничивать фактический платёж по телу долга:  
`principal = Math.min(principal, balance)`, и при необходимости скорректировать `payment` (например, `payment = principal + interest`), чтобы последняя строка отражала реальный платёж и нулевой остаток.

### 3.2. Дифференцированный + досрочные

- После досрочного погашения `balance` уменьшается, но `fixedPrincipalPortion` по-прежнему считается от исходного `loanAmount` и `numberOfPayments`.
- При большом досрочном может получиться `fixedPrincipalPortion > balance` → те же проблемы, что в п. 3.1.
- Логика «reduce term» / «reduce payment» в графике заточена под аннуитет (пересчёт через аннуитетную формулу). Для дифференцированного сценария «reduce payment» в реальности обычно означает пересчёт графика с новым остатком и тем же сроком (новый фиксированный principal = newBalance / remainingMonths). Сейчас после досрочного с типом reducePayment пересчитывается аннуитетный платёж — для чисто дифференцированного продукта это может расходиться с ожиданиями.

**Рекомендация:**  
- Всегда ограничивать `principal` текущим `balance`.  
- Для дифференцированного + reducePayment при желании заложить отдельную ветку: пересчёт графика с новым остатком и оставшимся сроком (фиксированный principal от newBalance).

## 4. Эффективная ставка (effectiveInterestRate)

- В `MortgageResults` и `MortgageCalculationResults` есть поле `effectiveInterestRate`.
- В `mortgageCalculator.ts` оно **нигде не вычисляется** и не возвращается (в return его нет).
- В `LocalMortgageService.calculateBase` в результат пробрасывается `result.effectiveInterestRate` → всегда `undefined`.

**Рекомендация:** либо убрать поле из типов/ответа, либо реализовать расчёт (например, через ставку, при которой NPV потока платежей равен сумме кредита) и возвращать его из ядра.

## 5. Оптимизация алгоритмов

### 5.1. Дифференцированный total cost в mortgageCalculator

- Сейчас: цикл по всем месяцам с вызовом `calculateMonthlyPayment(..., i)`.
- Можно заменить на формулу: сумма процентов по дифференцированному кредиту  
  `totalInterest = P * r/12 * (n + 1) / 2` (арифметическая прогрессия остатков),  
  тогда `totalCost = loanAmount + totalInterest`. Это O(1) вместо O(n).

### 5.2. originalTotalInterest в amortizationSchedule

- Для дифференцированного уже считается в цикле (строки 122–131) — один проход, без построения полного графика. Оставить как есть приемлемо; при желании можно заменить на ту же формулу суммы арифметической прогрессии, чтобы не дублировать логику с `financialMath`.

## 6. Поток данных: входящие и выходящие

### Входящие (форма → ядро)

- **Источник:** форма (`LoanDetailsType` — строки) → `unformatFormValues` → числа/даты.
- **В провайдер** попадает уже `LoanDetailsValues` (числа) при сабмите и при открытии расчёта из истории/результата (`payloadToFormValues` используется только для подстановки в форму при «редактировать»).
- В ядро уходят: `loanAmount`, `interestRate`, `loanTerm`, `startDate`, `paymentType`, `paymentDay`, опционально `earlyPayments`, `regularPayments`. Типы и обязательность совпадают с интерфейсами сервиса. **Поток входящих данных выстроен правильно.**

### Выходящие (ядро → UI)

- **Базовый расчёт:** `MortgageCalculationResults` (monthlyPayment, totalInterest, totalCost, payoffDate, loanTerm, paymentType, effectiveInterestRate) → сохраняется в `mortgageResults` в провайдере.
- **График:** `AmortizationScheduleResults` (schedule[], summary) → `amortizationResult`.
- Компоненты (`ResultsDisplay`, `PaymentSchedule`, графики) берут данные из контекста. Дублирования расчётов на UI нет. **Поток выходящих данных выстроен правильно.**

### Замечание по типам

- `AmortizationScheduleResult` (из `amortizationSchedule.ts`) и `AmortizationScheduleResults` (из `IMortgageService`) по полям совпадают; в `LocalMortgageService` делается приведение `result as AmortizationScheduleResults`. Имеет смысл оставить один общий тип (например, в `IMortgageService`) и использовать его в утилите, чтобы не держать два имени и каст.

## 7. Округление и точность

- Явного округления до копеек (2 знака) в ядре нет: везде работа с float.
- Для отображения округление делается, скорее всего, в компонентах (например, при форматировании). Для консистентности и избежания накопления ошибок в длинных графиках можно ввести округление в ядре (например, до 2 знаков при записи в строку графика и в итоговые суммы). Это не ошибка, а улучшение предсказуемости.
