# Обзор архитектуры и рекомендации

Документ фиксирует явные проблемы проектирования, возможные оптимизации и идеи для дальнейшего развития после рефакторинга потока данных.

---

## 0. Storage как единственный источник правды

**Принято:** Все данные расчёта живут только в storage (Telegram Cloud Storage или mock localStorage). Навигация не использует `location.state`.

- **Результат:** `/result?id=...` — id обязателен, данные всегда загружаются через `storage.getById(id)`.
- **Калькулятор (новый расчёт):** `/calculator` — форма пустая; по submit делается `storage.save(payload)` → редирект на `/result?id=${saved.id}`.
- **Калькулятор (редактирование):** `/calculator?id=...` — форма подставляется из `storage.getById(id)`; по submit — `storage.update(id, payload)` → редирект на `/result?id=${id}`.
- **Главная:** клик по расчёту → `navigate(\`/result?id=${calc.id}\`)`.
- **Кнопка «Изменить параметры» на результате:** `navigate(\`/calculator?id=${savedId}\`)`.

Плюсы: один источник правды, прямая ссылка на расчёт по id, нет потери данных при обновлении страницы или открытии в новой вкладке. Минус: при переходе на результат всегда есть короткая загрузка из storage (можно при желании смягчить кэшем).

---

## 1. Что сделано в рефакторинге

- **Домен** (`src/domain/`): единый источник типов и контрактов (CalculationData, SavedCalculation, type guards, generateCalculationId, ResultPageState, getSavedIdFromState).
- **Единая DTO**: форма и хранилище работают с CalculationData; при открытии расчёта данные подставляются один раз из SavedCalculation.
- **Убраны дубли**: один type guard, одна генерация id, один тип payload; типы не реэкспортируются из MortgageProvider.
- **Типизация навигации**: ResultPageState и getSavedIdFromState для чтения state при переходе на result.
- **Угловой кейс**: открытие по `?id=...` при удалённом расчёте — показ плейсхолдера «Расчёт не найден» и состояния загрузки при getById.
- **Провайдер**: один эффект для базового расчёта и графика (меньше лишних перезапусков).

## 2. Замечания по архитектуре

### 2.1 MortgageProvider — совмещение ролей

**Сейчас:** Провайдер хранит и вход (loanDetails, earlyPayments, regularPayments), и выход (mortgageResults, amortizationResult), и сам запускает расчёты в useEffect.

**Минусы:** Один контекст отвечает за «текущий расчёт» и за «результаты». При росте фич (кэш, несколько расчётов, офлайн) логику расчёта удобнее выносить в отдельный слой.

**Рекомендация (на будущее):** Вынести «состояние текущего расчёта» (inputs) и «результаты расчёта» в отдельные контексты или в хук `useCalculation(id | null)`, который по входным данным возвращает { results, schedule, loading, error }. Провайдер тогда только прокидывает данные и результаты, а вызов сервиса — в хуке или в отдельном CalculationRunner.

### 2.2 setMortgageResults / setAmortizationResult в контексте

**Сейчас:** Сеттеры результатов отдаются в контекст, но снаружи нигде не вызываются — только внутри провайдера в эффекте.

**Рекомендация:** Не отдавать их в value контекста, оставить только чтение mortgageResults и amortizationResult. Так контракт контекста проще и не создаётся впечатление, что результаты можно менять снаружи.

### 2.3 types/storage.ts как реэкспорт

**Сейчас:** `types/storage.ts` только реэкспортирует типы из `@/domain`.

**Рекомендация:** Для новых мест импортировать типы из `@/domain`. При желании со временем убрать реэкспорт из `types/storage.ts` и везде перейти на `@/domain`, чтобы была одна явная точка входа для доменных типов.

### 2.4 Генерация id в domain

**Сейчас:** `generateCalculationId()` лежит в domain.

**Замечание:** По строгой слоистости генерация id часто считается инфраструктурой (UUID, nanoid). Для текущего формата (timestamp + random) размещение в domain допустимо; при переходе на внешнюю библиотеку id разумно перенести в слой storage и вызывать оттуда.

### 2.5 Слабая проверка в isCalculationData

**Сейчас:** Type guard проверяет только наличие объекта и поля `loanDetails` типа object.

**Риск:** В state может попасть объект с другой структурой (например, устаревший формат или ошибка), и мы примем его за CalculationData.

**Рекомендация:** Либо усилить guard (проверка полей loanDetails.loanAmount, loanDetails.interestRate и т.д.), либо при восстановлении из storage/state прогонять данные через Zod-схему и при невалидности подставлять дефолты или показывать «Не удалось загрузить расчёт».

## 3. Возможные оптимизации

### 3.1 Дебаунс записи в storage на странице результата

**Сейчас:** При любом изменении loanDetails / earlyPayments / regularPayments вызывается `storage.update(savedId, ...)`.

**Идея:** Дебаунсить вызов update (например, 300–500 ms), чтобы при быстром редактировании в модалке не отправлять много записей подряд. Важно: при размонтировании страницы выполнить отложенный update сразу (flush).

### 3.2 Кэш результатов расчёта

При частом переключении между расчётами по id можно кэшировать результаты по ключу (id или хэш CalculationData), чтобы не пересчитывать один и тот же расчёт. Имеет смысл при появлении списка расчётов с превью или при тяжёлых расчётах.

### 3.3 Навигация по id

Навигация на результат и калькулятор уже идёт только по id в URL (`/result?id=...`, `/calculator?id=...`). Дополнительно можно ввести хелперы `navigateToResult(id: string)` и `navigateToCalculator(editId?: string)` для единообразия.

---

## 4. Конвенции после рефакторинга

- **Типы расчёта (domain):** импорт из `@/domain` (CalculationData, SavedCalculation, LoanDetailsValues, EarlyPayment, RegularPayment).
- **Типы формы (строки для инпутов):** `@/types/form` (LoanDetailsType и т.д.).
- **Навигация:** только по id в URL. Результат — `/result?id=...`, калькулятор (редактирование) — `/calculator?id=...`. Данные всегда из storage.

---

## 5. Что можно не менять (сознательный выбор)

- **Опечатка Mortage:** переименование в Mortgage затрагивает роуты, файлы, импорты; оставлено как есть, при необходимости вынести в отдельный рефакторинг.
- **Единый контекст расчёта:** текущая схема «один провайдер = один текущий расчёт» достаточна для текущего объёма; разделение на несколько контекстов/хуков — при росте требований.
- **Синхронное сохранение при изменении:** явное «Сохранить» не вводилось; неявное сохранение при изменении оставлено как продуктовое решение, при желании дополняется дебаунсом (см. п. 3.1).
