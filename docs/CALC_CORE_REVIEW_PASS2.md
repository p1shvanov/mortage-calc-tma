# Второй проход: потенциальные недоработки расчётов

## 1. Критично: выход из цикла при reduceTerm с положительным остатком

**Проблема:** Условие цикла `while (month <= remainingTerm && balance > 0.01)`. При досрочном «сокращение срока» мы пересчитываем `remainingTerm = month + estimatedRemainingPayments`. Формула оценки может дать заниженное число (округление, накопленная погрешность), и цикл завершится по `month > remainingTerm`, пока `balance` ещё > 0.01. В итоге график обрывается с ненулевым остатком.

**Решение:** Выходить из цикла только по погашению: `balance <= 0.01`, а `remainingTerm` использовать только как ориентир. Добавить верхнюю границу итераций (например, `numberOfPayments + 24`), чтобы исключить бесконечный цикл.

---

## 2. originalTotalInterest считается по другой методологии, чем график

**Проблема:** В `amortizationSchedule` значение `originalTotalInterest` считается по месячной ставке (равные месяцы): для дифференцированного — цикл с `monthlyRate`, для аннуитета — `originalMonthlyPayment * n - loanAmount`. При этом помесячные проценты в графике считаются по ACTUAL_365. В итоге «экономия» `totalSavings = originalTotalInterest - newTotalInterest` смешивает две методики и может быть неточной или вводить в заблуждение.

**Решение:** Считать `originalTotalInterest` так же, как накопленные проценты в графике без досрочных: либо прогонять тот же цикл без досрочных и брать сумму процентов, либо вынести общую функцию «сумма процентов за срок» по ACTUAL_365 и использовать её и для базового сценария, и для originalTotalInterest в summary.

---

## 3. Несколько досрочных на одну дату

**Проблема:** Досрочные складываются в `Map` по ключу `date`. При двух записях с одной датой в график попадёт только последняя: `earlyPaymentsByDate.set(payment.date, ...)` перезаписывает предыдущее.

**Решение:** По одной дате суммировать суммы: при наличии уже записанной досрочной по этой дате добавлять к сумме и оставлять один тип (например, последний или приоритетный).

---

## 4. Пересекающиеся регулярные платежи

**Проблема:** При двух регулярных платежах с пересекающимися периодами (одни и те же месяцы) для каждого месяца в пересечении к `extraPayment` добавляется избыток обоих. Фактически пользователь может иметь в виду один платёж в месяц, а не два.

**Решение:** Либо явно суммировать избытки (и задокументировать), либо считать только один регулярный платёж на дату (например, первый подходящий по периоду или с приоритетом по сумме). Текущее поведение лучше описать в UI/документации или изменить по продукту.

---

## 5. Дифференцированный + «уменьшить платёж»

**Проблема:** После досрочного с типом «уменьшить платёж» следующий платёж пересчитывается по аннуитетной формуле. Для изначально дифференцированного кредита банки часто пересчитывают график по тому же типу (новый фиксированный principal = остаток / оставшиеся месяцы). Сейчас модель переключается на аннуитет.

**Решение:** Либо оставить как есть и явно описать в интерфейсе, либо добавить ветку для дифференцированного: после досрочного пересчитывать фиксированную часть тела от нового остатка и оставшегося срока.

---

## 6. Нулевые/невалидные входы

**Проблема:** При `loanAmount <= 0` или `loanTerm <= 0` получаем `numberOfPayments = 0`, цикл не выполняется, график пустой. `calculateMonthlyPayment` при нулевом сроке даёт деление на ноль (знаменатель аннуитета = 0). Валидация формы может не пропустить такие значения, но ядро не защищено.

**Решение:** В начале `generateAmortizationSchedule` и `calculateMortgage` возвращать «пустой» результат или проверять входы и бросать/возвращать ошибку при `loanAmount <= 0`, `loanTerm <= 0`, `interestRate < 0`.

---

## 7. Дробный срок кредита (лет)

**Проблема:** `numberOfPayments = loanTerm * 12` при `loanTerm = 10.5` даёт 126 месяцев. Цикл и формулы это допускают, но форма, скорее всего, принимает только целые годы. Стоит убедиться, что схема валидации и подсказки в UI согласованы с ядром (дробные годы разрешены или нет).

---

## 8. Часовой пояс и даты

**Проблема:** `new Date(startDate)` и `toISOString().split('T')[0]` зависят от локального времени. При UTC дата может сдвинуться на день. Для «календарных» дат платежей лучше работать в локальной дате или явно в UTC и единообразно.

**Решение:** Либо парсить дату как локальную (YYYY-MM-DD без времени), либо везде использовать UTC и один способ форматирования. При необходимости добавить тесты на граничные даты (конец месяца, високосный год).

---

## 9. Порядок: regular vs early на одну дату

**Проблема:** Сначала применяется разовая досрочная по дате, затем в цикле по регулярным к `extraPayment` может добавляться избыток регулярного. Тип берётся от регулярного («has precedence»). Если на одну дату есть и разовая досрочная, и регулярная, суммы складываются — это разумно, но стоит явно задокументировать.

---
