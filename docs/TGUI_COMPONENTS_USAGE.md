# Использование компонентов Telegram UI в проекте

Обзор **каждого** компонента из [@telegram-apps/telegram-ui](https://tgui.xelene.me/?path=/docs/getting-started--documentation) и где его можно применить в нашем приложении (ипотечный калькулятор TMA).

---

## Blocks (блоки контента)

### Accordion
**Уже используется:** EarlyPaymentsForm, RegularPaymentsForm, RegularEarlyPaymentsForm — секции «Досрочные платежи» и «Регулярные платежи» раскрываются/сворачиваются.

**Где ещё применить:**
- **Результат (MortageResult):** свернуть «Сводка по платежам» (ResultsDisplay) в аккордеон, чтобы сначала был виден только заголовок и ключевая цифра (ежемесячный платёж), остальное по тапу.
- **Калькулятор (LoanForm):** опционально обернуть «Детали кредита» в Accordion с открытым состоянием по умолчанию для единообразия с остальными блоками.

### Avatar
**Не используется.**

**Где применить:**
- **HomePage:** если позже появится профиль/аккаунт — аватар пользователя в шапке или в настройках.
- Сейчас в приложении нет сущности «пользователь» — компонент можно не трогать.

### AvatarStack
**Не используется.**

**Где применить:**
- Нет сценария «несколько пользователей/аватаров» в текущем функционале. Оставить про запас (например, если появится «расчёт от нескольких человек»).

### Badge
**Уже используется:** в Accordion.Summary в формах досрочных/регулярных платежей — показывает количество добавленных пунктов (тип `dot`, `large`).

**Где ещё применить:**
- **HomePage, ячейка расчёта:** бейдж «новый» или «изменён» рядом с датой, если когда‑нибудь появятся статусы расчётов.
- **Результат:** бейдж «сохранено» рядом с заголовком сводки, если расчёт пришёл из сохранённого.

### Banner
**Не используется.**

**Где применить:**
- **HomePage или Calculator:** баннер-подсказка вверху страницы («Здесь можно сохранять расчёты» / «Досрочные платежи сокращают срок»).
- **Результат:** одноразовый баннер «Расчёт сохранён в облаке» после первого сохранения.
- **Калькулятор:** предупреждение «Заполните обязательные поля» над формой, если нужен явный CTA.

### Blockquote
**Не используется.**

**Где применить:**
- **FAQ / справка:** если появится экран с текстом правил или подсказок — цитаты из условий банка или выдержки из документации.
- **Результат:** выделить важную фразу типа «Экономия на процентах» или «Итого к возврату» как цитату.

### Button
**Уже используется:** везде — «Новый расчёт», «Рассчитать», «Назад», «Добавить досрочный платёж», «Удалить», «Готово», в Placeholder и т.д.

**Где ещё применить:**
- Уже покрыто; следить за единым использованием `size` (s/m/l) и `mode` (plain/outline/default) по гайдам tgui.

### Card
**Не используется.**

**Где применить:**
- **HomePage, список расчётов:** каждую запись можно оформить как Card вместо одной Cell — карточка с суммой, сроком, датой и кнопками «Открыть»/«Удалить».
- **Результат:** блок «Сводка по платежам» обернуть в Card для визуального выделения на фоне графика и таблицы.
- **ChartsContainer:** каждый график (Line, Bar, Pie, Doughnut, Radar) обернуть в Card с заголовком графика внутри.

### Cell
**Уже используется:** HomePage (список расчётов, настройки), ResultsDisplay (все строки сводки), DisplayData, формы (списки пунктов досрочных/регулярных платежей).

**Где ещё применить:**
- **PaymentSchedule:** строку таблицы можно дублировать или заменять компактным видом через Cell (например, в мобильном варианте «месяц — платёж — баланс» одной Cell на платёж с раскрытием по тапу).
- Уже основной строительный блок списков — продолжать использовать везде, где нужна строка с subhead/subtitle/before/after.

### ButtonCell
**Не используется.**

**Где применить:**
- **HomePage:** строка «Новый расчёт» внизу — заменить текущий `Button` в `Section` на `ButtonCell` внутри списка, чтобы строка выглядела как часть списка (как в нативных списках Telegram).
- **Настройки:** «Язык», «Тема», «FAQ» — оформить как `ButtonCell` с переходом/действием при тапе для единообразия с Cell.

### Info (Cell)
**Не используется.**

**Где применить:**
- **ResultsDisplay:** строки только для отображения (сумма, процент, срок, дата) — если в библиотеке Info даёт отдельный вид «информационная ячейка», использовать вместо обычной Cell для чисто информационных строк.
- **DisplayData:** строки с данными (init-data, theme params и т.д.) — тем же стилем Info.

### Navigation (Cell)
**Не используется.**

**Где применить:**
- **HomePage:** каждая сохранённая запись расчёта — тап ведёт на экран результата; оформить как Navigation Cell (стрелка/chevron справа).
- **Настройки:** «Язык», «Тема», «FAQ» — как навигационные ячейки к подэкранам или внешним ссылкам.
- **MortageResult:** кнопка «Изменить параметры» при отсутствии Main Button — можно оформить как Navigation Cell «К калькулятору».

### IconButton
**Уже используется:** PaymentSchedule — кнопка с иконкой статистики у каждой строки для открытия модалки с разбивкой платежа.

**Где ещё применить:**
- **HomePage, ячейка расчёта:** иконка «удалить» (корзина) и «открыть» (стрелка) вместо или рядом с текстовыми кнопками.
- **ResultsDisplay:** иконки вместо эмодзи (💰, 📈, ⏱️, 🏁, 💵) — использовать иконки из `telegram-ui/dist/icons` и IconButton или IconContainer для единого стиля.
- **ChartsContainer:** переключение типа графика или полноэкран — IconButton в углу карточки.

### IconContainer
**Уже используется:** BackButton, TabView (иконки вкладок «Графики» / «График платежей»).

**Где ещё применить:**
- Везде, где рядом с текстом нужна иконка из библиотеки (Cell before, Button before, заголовки секций).

### Image
**Не используется.**

**Где применить:**
- **EnvUnsupported / InitDataPage:** вместо голого `<img>` для стикера/иллюстрации использовать компонент Image из tgui (если он даёт обрезку, скругления, placeholder).
- Любые будущие иллюстрации (пустые состояния, ошибки) — через Image.

### InlineButtons
**Не используется.**

**Где применить:**
- **HomePage, ячейка расчёта:** блок «Открыть» + «Удалить» оформить как InlineButtons вместо div с двумя Button.
- **PaymentSchedule, модалка:** под графиком разбивки — «Закрыть» и при желании «Поделиться» через InlineButtons.
- **Placeholder (нет расчётов):** одна кнопка уже есть; при добавлении второй (например «Как пользоваться») — InlineButtons.

### List
**Уже используется:** на всех основных экранах как обёртка для Section/Cell (HomePage, LoanForm, MortageResult, формы).

**Где ещё применить:**
- Продолжать использовать как корневой контейнер списков; при появлении форм — `List Component='form'` как в LoanForm.

### Placeholder
**Уже используется:** HomePage (нет сохранённых расчётов), MortageResult (нет данных для результата), EnvUnsupported (старый клиент).

**Где ещё применить:**
- **Ошибка загрузки списка расчётов (HomePage):** при reject storage.getList() показывать Placeholder с текстом ошибки и кнопкой «Повторить».
- **PaymentSchedule при пустом графике:** если по какой-то причине schedule пустой после расчёта — Placeholder «Нет данных графика».
- **ChartsContainer:** при отсутствии данных — Placeholder «Введите параметры и нажмите Рассчитать».

### Section
**Уже используется:** повсеместно — заголовки и футеры блоков (история, настройки, детали кредита, досрочные/регулярные платежи, сводка, графики, таблица).

**Где ещё применить:**
- Любой новый блок контента оформлять через Section с header/footer при необходимости; уже основной каркас экранов.

### Steps
**Не используется.**

**Где применить:**
- **Калькулятор (LoanForm):** отобразить шаги «1. Детали кредита → 2. Досрочные платежи → 3. Регулярные платежи → 4. Результат» для наглядности прогресса.
- **Онбординг:** если появится пошаговое знакомство с приложением — Steps.

### Timeline
**Не используется.**

**Где применить:**
- **Результат / история платежей:** альтернативное представление графика платежей — Timeline по годам/месяцам с точками «Платёж», «Досрочное погашение», «Конец срока».
- **Досрочные платежи в форме:** список дат досрочных платежей как Timeline вместо только списка Cell.

## Feedback (обратная связь)

### CircularProgress
**Не используется.**

**Где применить:**
- **Загрузка расчёта (MortageResult по id из ссылки):** пока данные подтягиваются — CircularProgress в центре или в Placeholder.
- **Сохранение/удаление расчёта:** при долгом запросе к облаку — маленький CircularProgress рядом с кнопкой или в Snackbar.

### Progress
**Не используется.**

**Где применить:**
- **Результат:** линейный прогресс «Погашено X% от срока» или «Погашено X% от суммы долга» по данным amortization.
- **Калькулятор:** при желании — прогресс заполненности формы (сколько полей заполнено).

### Skeleton
**Уже используется:** HomePage (загрузка списка расчётов), LoadingFallback (роут), ChartsContainer (lazy-графики), PaymentSchedule (модалка с PieChart).

**Где ещё применить:**
- **MortageResult при загрузке по id:** пока расчёт грузится по searchParams.id — блок Section с 4–5 Skeleton вместо ResultsDisplay.
- **Открытие сохранённого расчёта с HomePage:** короткая подсветка Skeleton перед навигацией на результат (опционально).

### Snackbar
**Уже используется:** HomePage — «Удалено» после удаления расчёта.

**Где ещё применить:**
- **Калькулятор:** после успешного перехода на результат — «Расчёт сохранён» (если сохраняем в облако при сабмите).
- **Результат:** «Параметры обновлены» при возврате с калькулятора с изменёнными данными.
- **Ошибка сохранения/удаления:** Snackbar с текстом ошибки и кнопкой «Повторить» (Snackbar.Button).

### Spinner
**Уже используется:** в Input (валидация) при isValidating.

**Где ещё применить:**
- **Кнопка «Рассчитать» (fallback без Main Button):** пока isSubmitting — показывать Spinner внутри кнопки (у Button есть loading, но при необходимости можно дублировать Spinner в after).
- **Кнопки «Удалить» / «Сохранить»:** при запросе показывать Spinner в ячейке или в кнопке.

### Spoiler
**Не используется.**

**Где применить:**
- **Результат:** длинный текст вроде «Как мы считаем» или юридического дисклеймера — скрыть под Spoiler.
- **FAQ:** ответы на вопросы открывать по тапу через Spoiler.
- **PaymentSchedule:** детализация по месяцу (вместо модалки) — Spoiler под строкой с разбивкой «Основной долг / Проценты».

## Form (формы)

### Checkbox
**Уже используется:** DisplayData для отображения boolean (например в init-data/theme params).

**Где ещё применить:**
- **Калькулятор:** опция «Уведомлять о платежах», «Включить досрочные по умолчанию» и т.п., если появятся настройки расчёта.
- **Настройки:** «Тёмная тема», «Сохранять расчёты в облаке» — заменить переключатель темы на Checkbox или оставить Switch по дизайну.

### Chip
**Не используется.**

**Где применить:**
- **Тип платежа (аннуитет/дифференцированный):** вместо Select — два Chip «Аннуитет» / «Дифференцированный» с выбором одного.
- **Тип досрочного платежа (сократить срок / уменьшить платёж):** Chip для каждого пункта в списке досрочных.
- **Фильтр на HomePage:** «Все» / «За последний месяц» / «С сохранением» — Chip-фильтры.

### ColorInput
**Не используется.**

**Где применить:**
- Нет сценария выбора цвета в текущем продукте. Оставить для будущего кастомизации тем/графиков, если понадобится.

### FileInput
**Не используется.**

**Где применить:**
- Нет сценария загрузки файлов (импорт расчётов из CSV/Excel и т.д.). При появлении — использовать FileInput.

### Input
**Уже используется:** LoanDetailsForm (дата, обёртки), FormField, InputNumberFormat — суммы, проценты, срок, даты, тип платежа через Select.

**Где ещё применить:**
- Уже основа форм; при добавлении новых полей (примечание к расчёту, название) — те же Input/Select.

### Multiselect / Multiselectable
**Не используется.**

**Где применить:**
- **Калькулятор:** если появятся «месяцы с досрочным погашением» выбором нескольких дат из календаря или «выбор месяцев» — Multiselect.
- Пока сценарий один — оставить в планах.

### PinInput
**Не используется.**

**Где применить:**
- Нет ввода пин-кода/кода подтверждения в текущем сценарии. При появлении верификации — PinInput.

### Radio
**Не используется.**

**Где применить:**
- **Тип платежа (аннуитет/дифференцированный):** альтернатива Select — группа Radio в одной Section.
- **Тип досрочного:** «Сократить срок» / «Уменьшить платёж» — Radio в каждом пункте досрочных/регулярных платежей.

### Rating
**Не используется.**

**Где применить:**
- Нет сценария оценки (рейтинг приложения, отзыв). При появлении экрана обратной связи — Rating.

### Select
**Уже используется:** тип платежа, день платежа, тип досрочного/регулярного платежа.

**Где ещё применить:**
- Любые новые выпадающие списки (валюта, период отображения графика и т.д.).

### Selectable
**Не используется.**

**Где применить:**
- **Список расчётов (HomePage):** если нужен режим «выбрать несколько и удалить» — обернуть ячейки в Selectable и добавить кнопку «Удалить выбранные».

### Slider
**Не используется.**

**Где применить:**
- **Калькулятор:** срок кредита (лет) или процентная ставка — слайдер в дополнение к полю ввода для быстрого подбора.
- **Результат:** «Показать первые N месяцев» в таблице — Slider для N.

### Switch
**Не используется.**

**Где применить:**
- **Настройки (HomePage):** переключатель темы (светлая/тёмная) — заменить текущую кнопку на Switch для привычного вида настроек.
- **Калькулятор/настройки:** «Включить подсказки», «Сохранять в облаке» и т.п.

### Textarea
**Не используется.**

**Где применить:**
- **Калькулятор:** поле «Примечание к расчёту» при сохранении.
- **Обратная связь / FAQ:** форма обратной связи с полем сообщения.

## Layout

### FixedLayout
**Не используется.**

**Где применить:**
- **Калькулятор (без Main Button):** закрепить кнопку «Рассчитать» внизу экрана через FixedLayout, чтобы она всегда видна при скролле.
- **Результат:** при необходимости закрепить «Изменить параметры» внизу при длинном контенте.

### Tabbar
**Не используется.**

**Где применить:**
- **Главная навигация:** если приложение вырастет (Главная / Калькулятор / Сохранённые / Настройки) — Tabbar внизу вместо только кнопки «Новый расчёт» на HomePage.
- Сейчас маршрутов мало — Tabbar опционален; при 4+ основных разделах станет уместен.

## Misc

### Divider
**Не используется.**

**Где применить:**
- **Настройки (HomePage):** разделитель между «Язык» и «Тема», между «Тема» и «FAQ».
- **ResultsDisplay:** между блоками «Платёж / Проценты» и «Срок / Даты» для визуального разрыва.
- **Формы:** между секциями Accordion или между группами полей.

## Navigation

### Breadcrumbs
**Не используется.**

**Где применить:**
- **Результат:** «Главная → Калькулятор → Результат» вверху страницы для навигации (особенно если нет нативной Back Button).
- **Глубокие экраны:** при появлении подстраниц (например Настройки → Язык) — Breadcrumbs.

### CompactPagination
**Не используется.**

**Где применить:**
- **PaymentSchedule:** альтернатива текущей Pagination — более компактный вид «‹ 1/10 ›» при нехватке места.
- **Список расчётов (HomePage):** при большом количестве сохранённых расчётов — CompactPagination внизу списка.

### Link (tgui)
**Не используется.** Есть свой компонент Link (react-router + openLink для внешних).

**Где применить:**
- **DisplayData:** для строк типа link использовать Link из telegram-ui для единого стиля (цвет, подчёркивание).
- **FAQ / справка:** ссылки на внешние статьи — tgui Link с открытием через TMA openLink.
- Уточнить: наш Link уже обрабатывает внешние переходы; при необходимости заменить на tgui Link и там вызывать openLink.

### Pagination
**Уже используется:** PaymentSchedule — постраничный просмотр графика платежей (12 месяцев на страницу).

**Где ещё применить:**
- **Список расчётов (HomePage):** при пагинации сохранённых расчётов (если решим не показывать все сразу).
- **ChartsContainer:** если графиков много и показываем по одному с переключением — Pagination или SegmentedControl уже используются через TabView.

### SegmentedControl
**Уже используется:** TabView на MortageResult — вкладки «Графики» / «График платежей».

**Где ещё применить:**
- **Калькулятор:** переключатель «Аннуитет / Дифференцированный» или «Детали / Досрочные / Регулярные» в виде сегментов.
- **Результат:** переключатель «Графики / Таблица» уже по сути есть; при добавлении третьей вкладки (например «Сводка») — тот же SegmentedControl.

### TabsList
**Не используется.**

**Где применить:**
- **MortageResult:** альтернатива текущему SegmentedControl — TabsList для вкладок «Графики» и «График платежей», если нужен другой визуальный стиль вкладок (подчёркивание, полоска).
- Сравнить с SegmentedControl в Storybook и выбрать один вариант для единообразия.

## Overlays

### Modal
**Уже используется:** PaymentSchedule — модалка с разбивкой платежа (PieChart) по клику на строку.

**Где ещё применить:**
- **Подтверждение удаления:** перед удалением расчёта на HomePage — Modal «Удалить расчёт?» с кнопками «Отмена» / «Удалить».
- **Результат:** «Поделиться расчётом» — модалка с ссылкой и кнопкой «Копировать» / «Поделиться».
- **Калькулятор:** предупреждение при уходе с формы с заполненными полями («Сохранить?» / «Выйти») через Modal.

### Popper
**Не используется.**

**Где применить:**
- Выпадающие меню, календари, подсказки привязанные к кнопке/полю (если не использовать Tooltip). Tooltip часто реализован поверх Popper.

### Tooltip
**Не используется.**

**Где применить:**
- **PaymentSchedule, иконка статистики:** подсказка «Разбивка платежа» при наведении/долгом тапе на IconButton.
- **PaymentSchedule, звёздочка у регулярного платежа:** Tooltip «Регулярный платёж» у символа *.
- **Результат, иконки в сводке:** пояснение к иконкам (что значит каждая строка).
- **Калькулятор:** подсказки к полям «Процентная ставка», «День платежа» и т.д.

## Service

### AppRoot
**Уже используется:** Layout.tsx, EnvUnsupported — корень приложения с appearance и platform.

**Где ещё применить:**
- Один корень на приложение — уже правильно; при мульти-окружении (например отдельный AppRoot для ошибки) оставить как есть.

### Tappable
**Не используется.**

**Где применить:**
- Кастомные кликабельные блоки (например карточка графика с переходом в полноэкран) — обернуть в Tappable для правильного ripple/feedback. Cell и Button уже используют tappable-поведение; для своих блоков — Tappable.

### VisuallyHidden
**Не используется.**

**Где применить:**
- Скрытые подписи для доступности: заголовок секции, описание кнопки «Рассчитать», подпись к полю ввода — обернуть в VisuallyHidden для скринридеров, не меняя визуал.

### RootRenderer
**Не используется.**

**Где применить:**
- Обычно для порталов (модалки, тултипы). Если в документации tgui порталы рендерятся через RootRenderer — использовать по гайду библиотеки; иначе оставить как есть с Modal.

## Typography

### Caption
**Уже используется:** ошибки под Input/Select, заголовки колонок таблицы (Table.tsx), ErrorBoundaryFallback.

**Где ещё применить:**
- Подписи под графиками, пояснения под полями, мелкий текст в футере Section — везде, где нужен вторичный мелкий текст.

### Headline
**Не используется.**

**Где применить:**
- Заголовки экранов: «Калькулятор», «Результат», «Настройки» — если нужен уровень между LargeTitle и Title.
- Заголовки секций в Section (header) — при желании заменить строку на Headline для единой типографики.

### LargeTitle
**Не используется.**

**Где применить:**
- **HomePage:** главный заголовок «Ипотечный калькулятор» вверху страницы.
- **Результат:** крупный заголовок «Ваш платёж» с суммой monthly payment.
- Один крупный акцент на экран — не перегружать.

### Subheadline
**Не используется.**

**Где применить:**
- Подзаголовки секций (например «Детали кредита», «Досрочные платежи») — использовать Subheadline в Section header или внутри Card.
- Подпись под главным числом на результате («в месяц», «до конца срока»).

### Text
**Уже используется:** повсеместно — содержимое ячеек, подписи, кнопки.

**Где ещё применить:**
- Продолжать как основной компонент для абзацев и строк текста; при необходимости использовать weight (1/2/3) для акцентов.

### Title
**Не используется.**

**Где применить:**
- Заголовки карточек (например «Сводка по платежам», «График платежей») — Title внутри Section или Card.
- Крупные числа на результате можно обернуть в Title (например ежемесячный платёж).

### Typography
**Не используется напрямую.** Text, Caption, Title и т.д. наследуют от Typography.

**Где применить:**
- Кастомные варианты типографики (свой размер/цвет через props) — использовать базовый Typography с нужными пропсами, если в дизайне появятся отличия от готовых Text/Title/Caption.

| Категория   | Уже используем | Имеет смысл добавить |
|------------|----------------|-----------------------|
| Blocks     | Accordion, Badge, Button, Cell, IconButton, IconContainer, List, Placeholder, Section | Banner, Card, ButtonCell, InlineButtons, Navigation (Cell), Steps, Timeline |

| Form       | Checkbox, Input, Select | Chip, Radio, Slider, Switch, Textarea |
