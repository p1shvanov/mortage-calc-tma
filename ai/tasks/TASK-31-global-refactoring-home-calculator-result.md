# TASK-31: Глобальный рефакторинг — Home, Калькулятор, Результат

## Цель

Привести приложение к production-ready структуре: отдельная главная страница с историей и настройками, упорядоченная страница расчётов с разделением логики формы и калькуляторного ядра, страница результата с понятной навигацией и лучшими практиками UI/UX.

## 0. Хранение данных (история расчётов)

- **Production:** данные о расчётах хранятся в **Telegram Cloud Storage** (Telegram API). Все сохранённые расчёты — только из глобального хранилища.
- **Среда разработки:** Cloud Storage в dev может быть недоступен или неудобен — заводим **JSON-файл** и **мокаем** чтение/запись истории расчётов через него (например, `dev/mock-storage.json` или через отдельный storage adapter с реализацией для dev).

## 2. Страница калькулятора (форма) — порядок и разделение логики

### Ключевой принцип потока данных

- **Форма** собирает валидные данные и **передаёт их на страницу результата**. Форма не вызывает расчёт и не дергает провайдер расчётов.
- **Страница результата** получает данные (при монтировании) и **сама запускает расчёт**. В дальнейшем это позволит заменить локальный расчёт на серверный API: страница результата просто получит данные и отобразит результат (локально посчитанный или пришедший с сервера).

### Разделение ответственности

- **Форма (Form layer):** только ввод, валидация (Zod), преобразование в доменную модель. При сабмите — передаёт готовый объект данных на страницу результата (через навигацию). Без вызовов калькуляторного ядра и без логики «куда идти» внутри формы (это делает страница калькулятора).
- **Калькуляторное ядро (Core):** только бизнес-логика расчёта (utils + mortgage service). Вызывается со страницы результата (или в будущем — серверный API отдаёт готовый результат).
- **Страница калькулятора (Page):** при сабмите формы получает валидные данные → опционально сохраняет в историю (хранилище) → **navigate на `/result` с передачей данных** (см. ниже).
- **Страница результата:** при монтировании читает переданные данные → запускает расчёт (или в будущем запрос к API) → отображает результат.

### Передача данных: форма → страница результата

- **Рекомендация:** использовать **`location.state`** (react-router): при сабмите формы вызывать `navigate('/result', { state: { loanDetails, earlyPayments, regularPayments } })`. На странице результата в `useEffect` при монтировании читать `useLocation().state` и по этим данным вызывать расчёт (или устанавливать в провайдер и дать провайдеру посчитать). Плюсы: не раздуваем URL, типизация, подходит для сложных объектов.
- **Альтернатива (если нужны шерируемые ссылки):** `searchParams` — сериализовать параметры в query (например, base64 JSON или отдельные поля). В Telegram Mini App и HashRouter это возможно; минус — длинный URL. Можно добавить позже для «поделиться расчётом».
- **Итог:** на первом этапе — передача через **state**; при необходимости позже — дублировать опцию с searchParams для шаринга.

### Текущее состояние (кратко)

- Сейчас форма в `useLoanForm` при сабмите вызывает `setLoanDetails`/`setEarlyPayments`/`setRegularPayments` и `navigate('result')`, а расчёт триггерится в `MortgageProvider` по эффекту. Нужно изменить: форма только отдаёт данные; страница калькулятора при сабмите делает `navigate('/result', { state })`; страница результата при монтировании берёт данные из `location.state` и запускает расчёт (например, устанавливает в провайдер один раз при mount, либо вызывает сервис и держит результат локально).

### План рефакторинга

- [ ] **Форма:** вынести из `useLoanForm` навигацию и вызовы провайдера. Хук формы возвращает форму с `onSubmit`, который получает только валидные данные (доменная модель через `unformatFormValues`). Колбэк «что делать с данными» задаётся на странице калькулятора.
- [ ] **Страница калькулятора:** при сабмите — получить value из формы → при необходимости сохранить в хранилище (Telegram API / JSON-мок) → `navigate('/result', { state: { loanDetails, earlyPayments, regularPayments } })`.
- [ ] **Страница результата:** при монтировании — если есть `location.state` с данными расчёта, установить их в провайдер (или вызвать mortgageService и записать результат в локальный state/провайдер), затем отрисовать. Если state нет (прямой заход или переход из истории по id) — загрузить расчёт из хранилища по id и подставить в state/провайдер, затем отрисовать.
- [ ] **Ядро:** вызов расчёта только со страницы результата (или из провайдера по установленным данным). Форма не импортирует mortgage service.

### Итог по разделению

| Слой              | Ответственность |
|-------------------|------------------|
| Form              | Ввод, валидация, преобразование в доменную модель; при сабмите — отдаёт данные наружу. |
| Calculator page   | Сабмит → (опционально сохранить в хранилище) → navigate('/result', { state }). |
| Result page       | Монтирование → взять данные из state (или хранилища) → запустить расчёт → отобразить. В будущем — запрос к серверному API. |
| Calculator core   | Расчёт по параметрам (utils + mortgage service), вызывается со страницы результата. |

## 3. Страница результата — данные на монтировании, навигация и UI/UX

### Получение данных и расчёт

- Страница результата **получает данные при монтировании** и сама запускает расчёт (или в будущем — запрос к серверному API).
- **Источники данных:**
  1. **Переход с формы:** `location.state` с `{ loanDetails, earlyPayments, regularPayments }`. При mount — взять state, вызвать расчёт (через провайдер или напрямую mortgageService), отобразить.
  2. **Переход из истории (Home):** при клике на карточку расчёта — либо navigate на `/result` с `state` (если данные подгружены из хранилища), либо navigate на `/result?id=<savedId>` и на странице результата по `id` загрузить расчёт из хранилища (Telegram API / JSON-мок), затем запустить расчёт и отобразить.
- Если данных нет (прямой заход на `/result` без state и без id) — показать пустое состояние + кнопку «Перейти к расчёту» → `/calculator`.

### Требования по навигации и UI/UX

- **Возврат к форме:** кнопка «Назад» (Telegram Back Button через `Page back={true}`) ведёт на предыдущий экран (калькулятор или Home). При необходимости — явная кнопка «Изменить параметры» внизу.
- **Шапка:** использовать `Page back={true}`; при необходимости — заголовок секции «Результат расчёта» в контенте.
- **Иерархия:** сначала сводка (`ResultsDisplay`), затем вкладки (графики, таблица).
- **Пустое состояние:** нет данных → сообщение + кнопка «Перейти к расчёту» → `/calculator`.

### Технические шаги

- [ ] На монтировании: читать `location.state` или `searchParams.id`; если есть данные — установить в провайдер (или вызвать сервис) и триггерить расчёт; если есть только id — загрузить из хранилища, затем расчёт.
- [ ] Если данных и id нет — показывать пустое состояние + ссылку на `/calculator`.
- [ ] Оставить `Page back={true}`; при желании добавить кнопку «Изменить параметры».
- [ ] Порядок экранов: Home → Calculator → Result; Back с Result → Calculator; с Calculator Back → Home.
